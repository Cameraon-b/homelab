# Entry 11 — Unexpected Host Shutdown, Recovery Validation, and Resilience Planning

## Date
May 19, 2026

## Objective
Validate recovery behavior of the Nora hypervisor after an unexpected shutdown, verify core services and VM functionality, identify failed services, and document resilience improvements for future power outage scenarios.

## Environment

### Host
- **Hostname:** `nora`
- **OS:** Ubuntu Server
- **Role:** KVM/libvirt Hypervisor
- **Primary Services:** libvirtd, Cockpit, bridged networking (`br0`)

### Guest VMs
- **WIN-DC01** — Domain Controller / DNS / AD Services
- **WIN-CLIENT01** — Domain Workstation

### Management Access
- SSH
- Cockpit Web Dashboard
- TigerVNC
- RDP (for guest systems)

---

## Event Summary

An accidental shutdown occurred after the host was connected to the wrong power strip during an effort to reduce power consumption in the lab. The monitor was connected to the “always-on” strip, but the Nora host itself remained on a switchable strip and was unintentionally powered off.

This event also highlighted a realistic operational concern, as brief residential power outages during storms are common in the local area.

---

## Objective of Recovery Validation

After bringing the host back online, the following checks were performed:

- Verify host boot integrity
- Confirm Cockpit service behavior after reboot
- Validate libvirt/KVM functionality
- Check VM availability
- Review failed services
- Evaluate current resilience against unexpected power loss

---

## Work Completed

### 1. Host Recovery Validation

Successfully logged back into Nora after unexpected shutdown.

Confirmed:

- Host booted normally
- Filesystem mounted correctly
- Core services accessible
- No obvious corruption or boot issues observed

---

### 2. Cockpit Dashboard Validation

Checked Cockpit status:

    sudo systemctl status cockpit.socket

**Result:**

- Cockpit started automatically after host boot
- Remote dashboard access from Cass remained functional

Confirmed:

- Web management available
- VM management interface accessible
- Host metrics/logs visible

---

### 3. Virtualization Service Validation

Checked libvirt/KVM service:

    sudo systemctl status libvirtd
    virsh list --all

**Result:**

- libvirtd functional
- VM definitions intact
- Virtualization stack recovered successfully

---

### 4. Failed Service Review

Checked for failed services:

    systemctl --failed

**Result:**

Only failed service:

    systemd-networkd-wait-online.service

**Assessment:**

- Likely timeout related to bridged networking startup
- Non-critical because networking became functional after boot
- Common behavior in bridge/DHCP-based virtualized environments

Confirmed:

- Cockpit reachable
- Network connectivity functional
- No broader networking failure observed

---

### 5. Operational Lessons Learned

This event exposed a physical infrastructure oversight:

- Monitor connected to always-on strip
- Host connected to switchable strip
- Incorrect power segregation after lab reorganization

This created an accidental shutdown scenario and revealed the need for recovery automation.

---

## Planned Resilience Improvements

### BIOS Recovery Behavior

Planned next session:

Enable:

    Restore on AC Power Loss → Power On

**Purpose:**

- Automatically boot Nora after outage
- Remove dependency on manual physical intervention

---

### VM Autostart Configuration

Planned next session:

    virsh autostart WIN-DC01
    virsh autostart WIN-CLIENT01

**Purpose:**

- Automatically restore guest infrastructure after host reboot

---

### Physical Power Management Improvements

Planned:

- Move Nora to always-on strip
- Label critical power connections
- Separate optional vs persistent lab equipment
- Consider UPS battery backup in future

---

## Lessons Learned

Unexpected outages are valuable infrastructure tests.

This event demonstrated:

- Host recovery behavior
- Service persistence
- Dashboard availability
- Virtualization resilience
- Physical infrastructure weaknesses
- Need for automatic power recovery planning

It also reinforced an important infrastructure mindset shift:

Nora is no longer functioning like a typical desktop computer — it is becoming persistent lab infrastructure that should recover automatically after power interruption.

- Reboot and validate full recovery chain
