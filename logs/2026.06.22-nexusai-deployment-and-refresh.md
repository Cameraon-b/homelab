# AETHER Log #26 — NexusAI Becomes an AETHER Service

**Date:** June 22, 2026

## Summary

This AETHER work cycle focused heavily on NexusAI, but the larger milestone is broader than one application: AETHER gained a more mature internal service deployment workflow.

NexusAI moved from a local prototype into a deployed homelab service running on Nora, backed by Docker Compose, internal DNS, GitHub source control, database backups, and a manual GitHub Actions deployment path. The message and approval loop remained functional through the transition, and groundwork was added for a safe file-based agent bridge.

This marks a shift in the lab from “individual services running on machines” toward a more coordinated internal platform where services, documentation, backups, deployments, and AI agents can be managed with clearer boundaries.

## High-Level AETHER Milestone

AETHER now has a working pattern for developing and deploying an internal service:

```text
Develop locally
→ commit and push to GitHub
→ manually trigger GitHub Actions
→ Nora self-hosted runner executes deploy script
→ database backup
→ git pull
→ Docker rebuild/restart
→ validate service on internal DNS
```

This workflow was proven with NexusAI and can become the template for future AETHER applications.

## Agent Bridge Groundwork

Work began on the first safe version of the NexusAI agent bridge.

The current approach is **file-based bridge mode**, not direct autonomous execution.

The bridge-file model uses JSON handoff files:

```text
scripts/bridge_queue/request-message-<id>-<agent>.json
scripts/bridge_queue/response-message-<id>-<agent>.json
```

The intended flow is:

1. Worker reads one approved/unread delivered message.
2. Worker creates a bridge request JSON file.
3. A human or external agent runtime creates a matching response JSON file.
4. Worker detects the ready response file.
5. Worker posts the response back to NexusAI.
6. AI-to-AI replies still require approval.
7. Processed bridge files are archived.
8. Worker exits.

This is intentionally boring and inspectable. It creates a boundary between NexusAI as message transport and any future agent cognition layer.

## Agent Worker Updates

The scheduled agent worker now supports several reply modes:

```text
ack-only
template
bridge-file
agent-bridge
```

`agent-bridge` currently acts as a deprecated alias for `bridge-file`.

The updated worker includes safeguards:

* Processes at most one message per run
* Does not loop
* Does not execute commands
* Supports dry-run mode
* Creates bridge request files
* Validates bridge response files
* Requires `ready: true`
* Requires matching message ID
* Requires matching agent name
* Requires non-empty reply body
* Limits reply body length
* Prevents duplicate replies to the same parent message
* Archives processed bridge files
* Keeps AI-to-AI responses behind Cameron approval

## AETHER Safety Boundary

This work did not add command execution.

NexusAI still does not execute:

* Shell commands
* PowerShell commands
* SSH commands
* Docker commands from the UI
* Service restarts from the UI
* Automatic remediation
* Infrastructure changes

Current rule:

```text
Approval does not equal execution.
```

AETHER rule:

```text
No agent shall chown the kingdom.
```

## Deployment Lessons Learned

This cycle clarified that “updated” can mean several different things:

```text
Local development version
GitHub repository version
Nora repository version
Running Docker container version
Browser-cached UI version
```

A successful `git pull` does not always mean the running web app or browser view has changed. UI updates may require Docker rebuilds and a browser hard refresh.

Useful Nora checks:

```bash
cd /home/noratheredeemer/nexusai
git log --oneline -1
git status
sudo docker compose ps
sudo docker compose logs --tail=50 nexusai
```
