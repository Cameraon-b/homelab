### 3. GitHub Actions Deployment Workflow Added

A GitHub Actions self-hosted runner was installed on Nora.

The runner allows GitHub to trigger deployment work directly on Nora without exposing Nora to the public internet or opening SSH externally.

A manual GitHub Actions workflow was added for deployment. The current model is intentionally manual:

* Code is updated locally
* Changes are committed and pushed to GitHub
* Cameron manually runs the GitHub Actions workflow
* Nora’s runner executes the deploy script
* NexusAI pulls the latest code, backs up the database, rebuilds, and restarts

This creates a controlled deployment gate.

Current deployment flow:

GitHub → Manual Actions workflow → Nora self-hosted runner → `deploy.sh` → database backup → git pull → Docker rebuild/restart → NexusAI live on `nexus.aether.lab`

Auto-deploy on push has not been enabled. Manual deployment is the correct safety boundary at this stage.

### 4. Deployment Troubleshooting Completed

Several deployment issues were identified and resolved:

* The initial deploy script failed because the normal user did not have Docker socket permissions.
* The immediate fix was to use `sudo docker compose ...` inside the deploy script.
* Nora initially appeared to have the old version of the app even after pulling.
* The issue was resolved through a combination of confirming Git state, rebuilding Docker, and refreshing the browser/static assets.
* The updated NexusAI UI is now visible at `http://nexus.aether.lab`.

Current lesson learned:

A successful `git pull` does not always mean the browser is seeing the latest running app. For UI changes, verify Git commit, Docker rebuild, running container, and browser cache.

### 7. Agent Bridge v1 Work Began

Hermes was given a task list to implement the first safe version of the NexusAI agent bridge.

The chosen first version is **bridge-file mode**.

This mode does not directly connect NexusAI to live command execution or unrestricted agent automation. Instead, it creates a safe JSON handoff layer.

Expected bridge-file workflow:

1. Agent worker reads one approved/unread delivered NexusAI message.
2. Worker creates a request JSON file.
3. External human/agent/runtime creates a matching response JSON file.
4. Worker detects the response file on the next run.
5. Worker posts the response back to NexusAI.
6. AI-to-AI replies still require Cameron approval.
7. Request and response files are archived.
8. Worker exits.

This keeps the agent bridge visible, inspectable, and reversible.

### 8. Agent Worker Updated

The scheduled NexusAI agent worker now supports:

* `ack-only`
* `template`
* `bridge-file`
* `agent-bridge` as a deprecated alias for `bridge-file`
* `--bridge-dir`
* `--bridge-fallback`
* `--dry-run`

The script still processes at most one message per run and does not loop.

Important safeguards present in the worker:

* No command execution
* Dry-run support
* Processes at most one message per run
* Creates bridge request files
* Validates bridge response files
* Requires `ready: true`
* Requires matching `message_id`
* Requires matching `agent`
* Requires non-empty `reply_body`
* Limits reply body size
* Prevents duplicate replies to the same parent message
* Archives processed bridge files
* Keeps AI-to-AI replies behind the approval gate

This is the right first bridge because it separates NexusAI transport from agent cognition.

## Current Safety Posture

NexusAI still does not execute commands.

The following remain out of scope:

* Shell execution
* PowerShell execution
* SSH execution
* Docker control from the UI
* Service restart buttons
* Automatic remediation
* Secret management
* Unapproved AI-to-AI delivery
* Autonomous infrastructure changes

Core rule:

**Approval records permission only. Approval does not equal execution.**

AETHER rule:

**No agent shall chown the kingdom.**

## Lessons Learned

* Manual gates are useful, not annoying, at this stage.
* GitHub Actions works best here as a controlled deploy button, not automatic deployment.
* Docker rebuild and browser cache can make it look like old code is still running.
* The deploy target path must stay consistent in documentation: `/home/noratheredeemer/nexusai`.
* It is important to distinguish:

  * local development version
  * GitHub version
  * Nora repo version
  * running Docker container version
  * browser-cached UI version
* The file bridge is the right first step toward agent cognition because it is inspectable and does not bypass approval.
