## Homepage Became the Front Door

Homepage was deployed on Nora using Docker Compose.

Initial access by IP failed because Homepage rejected the request with a host validation error. The logs showed:

`Host validation failed for: 192.168.1.137:3000`

This was resolved by setting the `HOMEPAGE_ALLOWED_HOSTS` environment variable in the Docker Compose file.

Once fixed, Homepage became reachable locally and was later configured through internal DNS as:

`home.aether.lab`

Firefox was then configured to open directly to `home.aether.lab`.

This changed the daily workflow immediately.

Instead of opening a browser to the public web, the browser now opens directly into AETHER.

That makes the lab feel less like a side project and more like a working environment.

## Gitea Deployment

Gitea was deployed on Nora using Docker Compose.

Because Uptime Kuma was already using port `3001`, Gitea was assigned a different host port and then placed behind Nginx Proxy Manager.

The final internal URL became:

`git.aether.lab`

Gitea immediately felt familiar because it closely resembles GitHub.

This made it clear that AETHER now has a local equivalent for source control, repositories, issues, branches, and future internal collaboration.

## Repository Import

The existing GitHub homelab repository was imported into Gitea.

GitHub repository:

`https://github.com/Cameraon-b/homelab`

Gitea repository:

`http://git.aether.lab/NoraTheRedeemer/homelab`

On Reeba, the local repository already existed at:

`C:\FilesForNora\GitHub\homelab`

The local Git remotes were updated so Reeba could act as a bridge between GitHub and Gitea.

The working model became:

GitHub  
↓  
Reeba local repo  
↓  
Gitea

This confirmed that changes from GitHub can be pulled locally and then pushed into Gitea.

Long-term, Gitea will become the internal working source of truth, while GitHub remains the public portfolio and off-site publishing layer.

## Updated Architecture Thinking

Today clarified the concept of the AETHER service layer.

Rather than letting AI agents directly interact with public services or sensitive infrastructure, the safer model is:

AI agent  
↓  
AETHER internal service layer  
↓  
Human review / approval  
↓  
Real action

The internal service layer now includes:

- Homepage
- BookStack
- Gitea
- NexusAI
- Memos
- Uptime Kuma
- Nginx Proxy Manager

This gives AI agents a safer workspace.

They can draft documentation, suggest commits, create plans, summarize logs, and prepare changes inside AETHER without directly owning production infrastructure or external accounts.

The principle is:

Agents can propose.  
Humans approve.  
Systems record what happened.

## Lessons Learned

The biggest lesson today was that usability matters.

AETHER had many useful services before today, but they were still accessed individually. Homepage changed that.

Once `home.aether.lab` became the browser start page, AETHER stopped feeling like a collection of URLs and started feeling like an environment.

The second lesson was that local-first infrastructure becomes more powerful when services are connected.

Gitea by itself is useful.

BookStack by itself is useful.

Homepage by itself is useful.

But together, they create a workflow:

Open AETHER  
↓  
View services  
↓  
Edit documentation  
↓  
Commit to Gitea  
↓  
Publish to GitHub

The third lesson was that AI agents should work through controlled internal systems rather than directly touching external services or sensitive infrastructure.

This fits the original NexusAI philosophy:

No agent shall chown the kingdom.
