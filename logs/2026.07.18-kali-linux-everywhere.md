# AETHER Log #37 – Pan Joins the Domain

**Date:** July 18–19, 2026

Tonight began with a resource problem.

Reeba has gradually taken on two important roles inside AETHER.

It serves as my primary Kali Linux workstation for security assessments, reconnaissance, and testing. It also hosts Mira, who has become the knowledge keeper of AETHER and the conversational interface to the documentation preserved in BookStack.

Both roles are valuable.

The problem is that they cannot always happen comfortably at the same time.

When Reeba is being used heavily for Kali testing, Mira may be unavailable. When Mira is running and being used during infrastructure work, the same machine is no longer dedicated entirely to security testing.

The solution seemed obvious:

Separate the workloads.

What followed was not obvious at all.

---

## Separating Mira and Kali

The original plan was to move one of the workloads onto a Raspberry Pi.

The Pi would become a small, dedicated Linux system inside AETHER, freeing Reeba so that Mira and Kali could both remain available without competing for the same machine.

The idea made architectural sense.

A Raspberry Pi is compact, inexpensive to operate, and well suited for lightweight infrastructure roles. It seemed like the perfect place to create a separate Linux node without consuming additional resources on Nora or disrupting the rest of the lab.

The installation began successfully, but it quickly became apparent that the Pi did not have enough performance for the experience I wanted.

The system worked.

It was simply too slow.

Package installation took a long time. General responsiveness was poor. Even basic setup tasks felt unnecessarily heavy. It became clear that continuing would produce a technically functional machine that I would avoid using because every interaction would feel like a chore.

That was not the goal.

The purpose of separating the systems was to make AETHER easier to operate, not to create another bottleneck.

Rather than forcing the original plan, I changed direction.

---

## Pan Moves to Cass

Cass had available processing power and memory, making it a much better host for a virtual machine than the Raspberry Pi was for a full interactive Kali installation.

A new Kali Linux virtual machine was created on Cass.

The system inherited the name **Pan**.

At first, Pan's purpose was straightforward:

Create a dedicated Kali environment so Reeba would no longer have to serve both Mira and security testing at the same time.

Once the VM was running, however, the project expanded.

If Pan was going to become a permanent Linux workstation inside AETHER, it made sense to treat it like a real enterprise endpoint rather than another isolated local machine.

That led to a new objective:

**Join Pan to the AETHER.LAB Active Directory domain.**

This would allow Pan to authenticate using centralized domain identities instead of relying only on local Linux accounts.

The local administrator was intentionally named:

```text
localadmin
```

rather than `cameron`.

That naming decision came from an earlier lesson inside AETHER.

Using the same username for both a local account and a domain account creates unnecessary ambiguity. It becomes harder to tell which identity is being used, where authentication is occurring, and which permissions apply.

By naming the local account `localadmin`, the distinction would remain clear:

```text
localadmin
```

would represent the emergency or machine-specific administrative identity.

```text
AETHER\cameron
```

would represent the centralized domain identity.

That small decision would later make troubleshooting much easier.

---

## The First Networking Problem

Before Pan could communicate properly with Active Directory, it needed to exist directly on the AETHER network.

The first address it received was:

```text
10.0.2.15
```

That address immediately revealed the problem.

VirtualBox had configured the VM using NAT networking.

Under NAT, Pan could reach outward through Cass, but it was not functioning as a normal peer on the local network. For Active Directory integration, Pan needed reliable direct communication with WIN-DC01 and the rest of the AETHER infrastructure.

The VirtualBox adapter was changed from NAT to **Bridged Adapter** mode.

After rebooting, Pan received:

```text
192.168.1.86
```

Pan was now directly present on the AETHER LAN.

Communication with other systems could be tested normally, and the VM could be monitored like any other infrastructure node.

Pan was added to Uptime Kuma.

The newest member of AETHER now had a heartbeat.

---

## Active Directory Begins with DNS

The next issue involved DNS.

Pan initially received its DNS configuration from the normal network infrastructure. That was sufficient for internet access, but it was not sufficient for Active Directory.

A domain-joined machine must be able to locate domain services through the DNS records hosted by the domain controller.

Pan's DNS server was changed to:

```text
192.168.1.15
```

the address of WIN-DC01.

After applying the change, the domain could be resolved:

```bash
nslookup aether.lab
```

The required Active Directory service records were also checked:

```bash
nslookup -type=SRV _ldap._tcp.dc._msdcs.aether.lab
```

and the Kerberos records were verified as well.

These tests confirmed that Pan could discover:

- The domain
- The domain controller
- LDAP services
- Kerberos services

This reinforced one of the most important lessons of the entire process:

**Active Directory integration is a DNS problem before it is an authentication problem.**

Without correct domain DNS, the rest of the configuration does not matter.

---

## The Expected SSSD Path

The original domain-join plan was based on the method already used successfully with AlmaLinux.

That approach relied on:

- `realmd`
- `adcli`
- `SSSD`
- `libnss-sss`
- `libpam-sss`

This is the modern Linux domain-integration path and seemed like the obvious choice.

The first sign of trouble appeared during package installation.

Kali could not locate:

```text
sssd
```

It also could not locate:

```text
sssd-tools
libnss-sss
libpam-sss
```

At first, this looked like a repository problem.

The Kali package sources were checked.

The rolling repository configuration was reviewed.

Package indexes were updated.

The expected packages still did not exist.

This was particularly confusing because `realmd` was available and could successfully discover the domain, yet the authentication backend it expected was missing.

The system could recognize that AETHER.LAB existed.

It simply could not install the components normally used to join it.

The current Kali release did not provide the required SSSD packages.

Rather than pulling packages from an unrelated Debian repository, mixing distributions, or introducing unsupported dependencies, the decision was made to abandon the SSSD path entirely.

A different integration method would be required.

---

## Moving to Samba and Winbind

The alternative was Samba with Winbind.

Winbind allows Linux systems to communicate with Active Directory and resolve Windows domain users and groups into Linux-compatible identities.

This was not the path originally planned, but it was a legitimate and widely used method.

The change introduced several additional moving parts:

- Samba configuration
- Kerberos configuration
- Winbind services
- NSS integration
- PAM integration
- UID and GID mapping
- Home-directory creation

The setup would be more manual than the SSSD approach, but it also provided a much deeper view into how the integration actually worked.

---

## Fixing the Hostname

Before proceeding, the machine hostname was standardized.

The original hostname used an uppercase form:

```text
Pan
```

It was changed to:

```text
pan
```

The machine was rebooted so the hostname would be applied consistently across the operating system and network services.

This avoided introducing another unnecessary variable into an already complex process.

---

## Kerberos Works Before Samba Does

Kerberos was tested independently before attempting the full domain join.

The following command was used:

```bash
kinit Administrator
```

The command succeeded.

A valid Kerberos ticket could then be displayed with:

```bash
klist
```

This was an important checkpoint.

It proved that:

- Pan could resolve the domain
- Pan could reach WIN-DC01
- Time synchronization was close enough for Kerberos
- The Administrator credentials were accepted
- The Kerberos realm configuration was functional

The domain controller and the Linux client were capable of authenticating with each other.

The remaining issue had to exist somewhere else.

---

## Samba Still Thought It Was in WORKGROUP

The next diagnostic command was:

```bash
sudo net ads info
```

Instead of returning information about AETHER.LAB, Samba behaved as though Pan belonged to:

```text
WORKGROUP
```

That single word became the most important clue of the domain-join process.

The default Samba configuration still contained:

```ini
workgroup = WORKGROUP
```

Samba was doing exactly what it had been configured to do.

Kerberos understood AETHER.LAB.

DNS understood AETHER.LAB.

Samba did not.

The Samba configuration at:

```text
/etc/samba/smb.conf
```

was edited carefully rather than completely replaced.

The domain settings were changed to:

```ini
workgroup = AETHER
realm = AETHER.LAB
security = ADS
kerberos method = secrets and keytab
```

This told Samba that Pan was not part of a standalone Windows workgroup.

It was intended to become an Active Directory domain member.

---

## Identity Mapping

Windows domain identities do not naturally use Linux UID and GID numbers.

Winbind therefore needed rules describing how Active Directory users and groups should be represented inside Linux.

The following mapping configuration was added:

```ini
idmap config * : backend = tdb
idmap config * : range = 3000-7999

idmap config AETHER : backend = rid
idmap config AETHER : range = 100000-999999
```

The wildcard mapping provided a small default range for non-domain identities.

The AETHER mapping used the RID backend to generate consistent Linux UID and GID values from Active Directory relative identifiers.

The default shell and home-directory path were also configured:

```ini
template shell = /bin/bash
template homedir = /home/%U
```

One subtle issue appeared here.

Several of the relevant default lines in the Samba configuration were preceded by semicolons.

They looked present at a glance, but they were actually comments and therefore inactive.

The necessary lines had to be uncommented or rewritten before Samba would use them.

This was another reminder that configuration files can be misleading when they contain examples that appear active but are not.

---

## Samba Finally Sees AETHER

After correcting the Samba configuration, the following command was run again:

```bash
sudo net ads info
```

This time, it returned valid information about the domain, including:

- LDAP server
- LDAP server name
- Realm
- Bind path
- LDAP port
- KDC server
- Server time
- Workgroup

Samba could now see the same environment that DNS and Kerberos had already been seeing.

The layers were finally aligned.

---

## Joining the Domain

The actual domain join was performed with:

```bash
sudo net ads join -U Administrator
```

The result was:

```text
Joined 'PAN' to dns domain 'aether.lab'
```

After hours of troubleshooting, Pan had successfully joined Active Directory.

Winbind was then enabled and started:

```bash
sudo systemctl enable --now winbind
```

The join was tested with:

```bash
sudo net ads testjoin
```

The result:

```text
Join is OK
```

The connection to the domain controller was verified with:

```bash
wbinfo --ping-dc
```

Winbind successfully contacted the NETLOGON service.

Domain users were enumerated with:

```bash
wbinfo -u
```

Domain groups were enumerated with:

```bash
wbinfo -g
```

The output included AETHER users such as:

```text
AETHER\cameron
```

along with the other domain accounts and security groups created throughout the evolution of the lab.

At this point, Pan was unquestionably a domain member.

However, being joined to the domain did not yet mean a domain user could successfully open a desktop session.

---

## Identity Resolution Works

Before attempting another graphical login, Linux identity resolution was tested directly.

The following command:

```bash
id AETHER\\cameron
```

returned a valid domain identity with a mapped Linux UID, primary group, and group memberships.

The account was assigned:

```text
uid=101103
```

The passwd database was also queried:

```bash
getent passwd AETHER\\cameron
```

The result included:

```text
/home/cameron
```

as the expected home directory.

This confirmed that:

- NSS could resolve the domain account
- Winbind could retrieve domain membership information
- UID and GID mapping worked
- The account had a valid Linux shell
- Linux knew where the user's home directory should be

The identity existed from Linux's perspective.

The next step was to log in.

---

## The Black-Screen Login Loop

The graphical login screen accepted:

```text
AETHER\cameron
```

The screen went black briefly.

Then it returned to the login screen.

At first, this looked like an authentication failure.

The logs told a different story.

Winbind reported:

```text
user 'AETHER\cameron' granted access
```

PAM reported that the session had opened successfully.

The password was correct.

Active Directory had authenticated the user.

Winbind had authorized the user.

Linux had begun starting the session.

The failure occurred afterward.

The logs contained errors showing that the desktop environment could not write files such as:

```text
/home/cameron/.Xauthority
```

and other files beneath:

```text
/home/cameron/.local/
```

The reason was simple:

```text
/home/cameron
```

did not exist.

Authentication was working perfectly.

The desktop session had nowhere to store the user's files.

---

## PAM Claimed Home Creation Was Enabled

The PAM configuration utility was opened using:

```bash
sudo pam-auth-update
```

The interface showed several enabled modules, including:

- Unix authentication
- Winbind NT/Active Directory authentication
- Session registration
- GNOME Keyring
- Create home directory on login

Based on the interface, automatic home-directory creation appeared to be enabled.

Yet the directory had not been created.

The actual PAM session configuration was checked:

```bash
grep mkhomedir /etc/pam.d/common-session
```

The command returned nothing.

The checkbox was enabled in the configuration interface, but the corresponding PAM line did not exist in the active configuration.

This was one of the strangest and most frustrating issues of the night.

The graphical configuration indicated one state.

The actual file reflected another.

---

## The Missing Line

The required line was added manually to:

```text
/etc/pam.d/common-session
```

The line was:

```text
session required pam_mkhomedir.so skel=/etc/skel/ umask=0022
```

The configuration was verified with:

```bash
grep mkhomedir /etc/pam.d/common-session
```

This time, the command returned:

```text
session required pam_mkhomedir.so skel=/etc/skel/ umask=0022
```

One missing PAM line had been the final barrier between successful authentication and a usable desktop session.

---

## A Domain Login on Kali

The login was attempted again using:

```text
AETHER\cameron
```

This time, the desktop opened.

The home directory was created automatically.

Inside the terminal, the prompt displayed:

```text
┌──(AETHER\cameron㉿pan)-[~]
└─$
```

The final verification command was:

```bash
whoami
```

The result:

```text
AETHER\cameron
```

Pan was no longer merely connected to the domain.

It was actively authenticating and running a Linux desktop session using an AETHER Active Directory account.

---

## Two Linux Domain-Integration Models

AETHER now contains two Linux systems joined to the same Active Directory domain through two different methods.

**Alma** uses:

- realmd
- SSSD
- standard modern enterprise Linux integration

**Pan** uses:

- Samba
- Winbind
- explicit identity mapping
- manual PAM configuration

What began as an unexpected package limitation ultimately expanded the educational value of the environment.

If Kali had provided SSSD normally, the domain join may have been much quicker.

It also would have revealed far less about the underlying process.

The Winbind path required understanding each layer directly:

- How DNS locates domain services
- How Kerberos validates identity
- How Samba becomes an AD member
- How Winbind retrieves users and groups
- How Windows identities become Linux UID and GID values
- How NSS exposes those identities to the operating system
- How PAM authenticates users and creates sessions
- How home directories are provisioned on first login

The difficulty forced a deeper understanding of the architecture.

---

## Lessons From the Process

This was one of the most challenging technical problems completed in AETHER so far.

It was difficult not because of one catastrophic failure, but because several smaller issues existed across multiple layers.

The process included:

- A Raspberry Pi that lacked the resources for the intended workload
- A redesign that moved Pan to a VM on Cass
- VirtualBox defaulting to NAT instead of bridged networking
- Incorrect DNS for Active Directory discovery
- Missing SSSD packages in Kali
- `realmd` discovering the domain but lacking its expected backend
- Samba retaining the default `WORKGROUP` configuration
- Commented idmap examples that appeared active at first glance
- Manual UID and GID mapping
- Winbind service configuration
- Successful authentication followed by a failed desktop session
- A missing home directory
- A PAM configuration interface that claimed `mkhomedir` was enabled when the actual configuration line was absent

No single fix solved everything.

The only workable approach was to test one layer at a time.

First networking.

Then DNS.

Then service discovery.

Then Kerberos.

Then Samba.

Then the domain join.

Then Winbind.

Then NSS.

Then PAM authentication.

Then session creation.

Then the home directory.

Each successful test eliminated another category of possible failure.

By the time the graphical login loop occurred, the logs made it possible to say with confidence that authentication was not the problem. That narrowed the investigation to session initialization and quickly exposed the missing home directory.

This was troubleshooting as a process of reducing uncertainty.

---

## Infrastructure Evolves Through Constraints

The most interesting part of this project is that none of it began with the goal of joining Kali to Active Directory.

It began because Reeba could not conveniently serve Mira and Kali at the same time.

That led to the Raspberry Pi.

The Raspberry Pi's limited performance led to a VM on Cass.

The VM became a permanent workstation.

The permanent workstation created an opportunity for centralized identity.

That opportunity led into DNS, Kerberos, Samba, Winbind, NSS, and PAM.

A practical resource constraint ultimately produced one of the deepest infrastructure exercises AETHER has experienced.

This is increasingly how AETHER evolves.

One problem exposes another opportunity.

A workaround becomes an architectural improvement.

A failed plan becomes a better design.
