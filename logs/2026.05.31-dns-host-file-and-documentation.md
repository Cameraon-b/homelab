## Work Completed

### HTTP Monitor Investigation

Observed intermittent failures affecting:

* BookStack HTTP Monitor
* Kuma HTTP Monitor
* Nginx HTTP Monitor

Symptoms included:

```text
getaddrinfo ENOTFOUND wiki.aether.lab
```

Monitors would periodically recover before failing again.

---

### DNS Resolution Troubleshooting

Investigation determined:

* Services remained online
* Reverse proxy remained operational
* DNS records existed and were correct
* Failures were isolated to DNS resolution from within Docker containers

Root Cause:

```text
Uptime Kuma Container
↓
Docker DNS Resolver
↓
Intermittent DNS Resolution
↓
Monitor Failures
```

The issue was not caused by BookStack, Nginx Proxy Manager, or DNS records themselves.

---

### Docker DNS Configuration

Configured the Uptime Kuma container to use WIN-DC01 directly for DNS resolution.

Added to Docker Compose:

```yaml
dns:
  - 192.168.1.15
```

Restarted the container stack.

Result:

* Stable DNS resolution
* Stable HTTP monitoring
* Elimination of intermittent ENOTFOUND errors

---

### Internal Service Monitoring

Successfully transitioned monitoring from IP-based checks to DNS-based service monitoring.

Current monitoring validates:

```text
DNS
↓
Reverse Proxy
↓
Application
```

rather than:

```text
IP
↓
Port
```

This provides improved visibility into service health.

---

### Windows Hosts File Integration

Objective:

Provide access to internal DNS names from Cass without modifying router DHCP or workstation DNS configuration.

Modified:

```text
C:\Windows\System32\drivers\etc\hosts
```

Added:

```text
192.168.1.137 wiki.aether.lab
192.168.1.137 kuma.aether.lab
192.168.1.137 test.aether.lab
```

Flushed DNS cache:

```powershell
ipconfig /flushdns
```

Verification:

```powershell
ping wiki.aether.lab
```

Successfully resolved:

```text
192.168.1.137
```

Result:

Cass can now access:

* wiki.aether.lab
* kuma.aether.lab
* test.aether.lab

without changing DNS settings or disabling IPv6.

### Architecture Decisions

Evaluated several approaches:

#### Option 1

Hosts file entries on individual systems

#### Option 2

Configure Cass to use WIN-DC01 for DNS

#### Option 3

Configure router DHCP to distribute WIN-DC01 as the primary DNS server

Decision:

Continue using the hosts file approach for Cass.

Reasoning:

* Low risk
* Easily reversible
* Minimal infrastructure dependency
* Appropriate for current lab size

## Lessons Learned

### Containers Have Their Own DNS Perspective

A host successfully resolving DNS does not guarantee that Docker containers will resolve DNS identically.

DNS behavior must be verified within the container itself when troubleshooting containerized applications.

---

### DNS Is a Dependency Chain

Application monitoring depends on:

```text
Application
↓
Reverse Proxy
↓
DNS
↓
Resolver
```

Failures may originate far below the application layer.

---

### Hosts Files Remain Useful

For small environments, hosts files provide a simple and effective alternative to centralized DNS deployment.

Benefits:

* Easy implementation
* Easy rollback
* Minimal risk

Limitations:

* Manual maintenance
* Poor scalability
