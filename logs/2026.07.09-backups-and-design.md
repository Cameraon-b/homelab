# Nexus Product Direction

The refined Nexus MVP remains centered on a simple product goal:

> Create one local-first workspace where humans and AI participants can communicate, collaborate, and build shared context without manually relaying messages between separate tools.

The MVP is not centered on tasks, forms, project boards, or enterprise workflow.

Its primary capabilities are:

* Main public conversation
* Topical conversations
* Direct messages
* Human-to-human communication
* Human-to-agent communication
* Agent-to-agent communication
* Participant directory
* Agent message drafts
* Admin approval of governed outbound messages
* Configurable participant policies
* Administrative oversight and audit history

Governance remains important, but it supports conversation rather than replacing it.

# Database Schema v1.1

The first complete implementation-oriented database model for Nexus reached an initial stable state.

Core tables include:

* `accounts`
* `participants`
* `account_participants`
* `conversations`
* `conversation_participants`
* `messages`
* `message_drafts`
* `approval_decisions`
* `participant_policies`
* `participant_permissions`
* `moderation_events`
* `audit_events`
* `sessions`
* `integration_tokens`

Several refinements were made after architectural review.

Notable improvements included:

* Separating accounts from visible workspace participants.
* Separating pending drafts from delivered messages.
* Removing redundant bidirectional draft/message references.
* Normalizing participant permissions rather than storing policy in an opaque JSON object.
* Simplifying conversation types to `main`, `topical`, and `direct`.
* Establishing one final approval decision per draft for the MVP.
* Separating moderation history from current participant state.
* Adding unique token constraints for human sessions and agent integrations.
* Treating audit events as historical records rather than authoritative domain state.

The database model now reflects the product's real center of gravity:

```text
Participants
      ↓
Conversations
      ↓
Messages
      ↓
Governance where required
```

The directory displays:

* Display name

Participant profiles provide additional information such as:

# Governance and Approval Design

The Draft Approval Queue was designed as a dedicated review workspace rather than a modal or small notification panel.

The screen includes:

* Pending draft queue
* Draft author
* Destination conversation
* Submission time
* Wait duration
* Surrounding conversation context
* Original draft
* Edited version
* Review notes
* Approval actions

Supported outcomes include:

* Approve
* Approve with edits
* Reject

The review interface also makes clear that an edited agent response remains agent-originated while recording that an administrator modified it before release.

Governance is therefore visible and auditable without becoming the dominant experience of the application.

# Empty and First-Run States

The wireframe set also intentionally covered situations where Nexus contains little or no activity.

These states include:

* Fresh installation and initial administrator creation
* Empty workspace
* Empty approval queue
* No direct messages
* No connected participants

The first-run workflow establishes a path from installation to meaningful collaboration:

```text
Install Nexus
      ↓
Create Admin Account
      ↓
Open Workspace
      ↓
Connect Participants
      ↓
Receive First Agent Draft
      ↓
Approve Draft
      ↓
First Message Appears
```

The intended result is that the workspace begins to feel alive within the first few minutes of operation.

# Virtual Machine Definitions

The current libvirt inventory was captured with:

```bash
sudo virsh list --all \
  > /mnt/backupdrive/AETHER/backups/manual/2026-07-07/inventory/virsh-list-all.txt
```

Each virtual machine definition was then exported:

```bash
for vm in $(sudo virsh list --all --name); do
  sudo virsh dumpxml "$vm" \
    > "/mnt/backupdrive/AETHER/backups/manual/2026-07-07/libvirt/xml/$vm.xml"
done
```

The resulting XML definitions preserve configuration such as:

* CPU allocation
* Memory allocation
* Virtual disk paths
* Virtual network interfaces
* Boot configuration
* Machine type
* Device assignments

Full VM disk images were intentionally excluded from this snapshot.

Existing VM image backups remain valid because the virtual machines have changed minimally since the earlier backup, aside from minor account and user changes.

VM disk images will therefore be backed up separately after significant VM changes or on a lower-frequency schedule.

# Backup Manifest

A Markdown manifest was added to the backup snapshot:

```text
/mnt/backupdrive/AETHER/backups/manual/2026-07-07/MANIFEST.md
```

The manifest documents:

* Backup date
* Included data
* Excluded data
* Purpose of the snapshot
* Recovery limitations

The manifest was created with a shell heredoc and `tee`:

```bash
cat <<EOF | sudo tee /mnt/backupdrive/AETHER/backups/manual/2026-07-07/MANIFEST.md
# AETHER Manual Backup

Date: 2026-07-07

## Included

- Docker Compose project folders
- Docker named volume data
- Host /etc snapshot
- Docker inventory files
- Host/storage inventory files
- libvirt VM XML definitions

## Not Included

- Full VM disk images
- Full host disk image
- Automated database dumps

## Notes

This backup captures service configuration, selected persistent Docker data,
host configuration, and VM definitions. It is intended as a lightweight
recovery snapshot, not a complete bare-metal backup.
EOF
```

The command only creates a text document describing the backup. It does not modify service data or system configuration.

# Backup Strategy Going Forward

The new operational snapshot establishes the template for future AETHER backups.

The planned process is:

```text
Inspect Environment
        ↓
Mount Backup Drive
        ↓
Create Dated Snapshot
        ↓
Copy Compose Projects
        ↓
Copy Persistent Volumes
        ↓
Capture Host Configuration
        ↓
Export VM Definitions
        ↓
Generate Inventories
        ↓
Create Manifest
        ↓
Verify Snapshot
        ↓
Safely Unmount Drive
```

Backups will remain manually initiated for now.

The goal is not to introduce fully unattended automation immediately.

Instead, the next step will be **manual-triggered scripting**.

# Planned Backup Scripts

Three scripts are planned for later development.

## `aether-backup-inspect.sh`

This script will check:

* Whether the backup drive is mounted
* Available disk space
* Active Compose projects
* Existing Docker volumes
* Virtual machine inventory
* Expected source directories

## `aether-manual-backup.sh`

This script will:

* Generate the current date
* Create the snapshot directory
* Copy Compose projects
* Copy selected Docker volumes
* Back up `/etc`
* Export libvirt XML
* Generate inventories
* Create the manifest
* Record execution output

## `aether-backup-verify.sh`

This script will validate:

* Expected directories exist
* Required inventory files exist
* Manifest exists
* VM XML files were exported
* Backup size is within an expected range
* Important files are not empty

The intended workflow will resemble:

```bash
./aether-backup-inspect.sh
sudo ./aether-manual-backup.sh
./aether-backup-verify.sh 2026-07-07
```

These scripts may eventually live in:

```text
aether-ops/
└── scripts/
    └── backups/
        ├── aether-backup-inspect.sh
        ├── aether-manual-backup.sh
        └── aether-backup-verify.sh
```

This approach preserves human control while making the operation repeatable.

It reflects the AETHER operating principle:

> Humans approve. Scripts execute. Systems record.

Today's work connected two different forms of engineering design.

Nexus design focused on how participants communicate.

Backup design focused on how infrastructure survives.

Both efforts relied on the same principles:

* Explicit structure
* Clear responsibility
* Human oversight
* Repeatable processes
* Durable records
* Intentional simplicity

The design packet explains Nexus.

The database schema defines its internal structure.

The wireframes describe how it will feel to use.

The backup snapshot preserves the environment that will host it.

Together, these artifacts reduce reliance on memory and improvisation.

# Lessons Learned

Several important lessons emerged.

* Wireframes reveal product and architecture decisions that prose alone cannot.
* Database relationships should be simplified before implementation begins.
* Participants should feel like members of the workspace, not hidden integrations.
* Backups must preserve recoverability, not merely copy files.
* Compose definitions, persistent data, host configuration, VM definitions, and inventories serve different recovery purposes.
* A backup should document what it contains and what it excludes.
* Manual-triggered automation is an appropriate intermediate step before unattended scheduling.
* Design work can feel slow, but diagrams and wireframes make the system tangible.
