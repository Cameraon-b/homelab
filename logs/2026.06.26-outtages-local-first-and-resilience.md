# AETHER Log #28 — Internet Outage, Local-First Infrastructure, and the Next Layer of Resilience

**Date:** June 26, 2026

## Summary

Today's internet outage unexpectedly became an architectural exercise. Essentially a whole day without access to the internet.

While the immediate inconvenience was obvious—no external websites, APIs, or cloud AI services—it quickly revealed something much more interesting. AETHER continued functioning internally in ways that I had not fully appreciated before.

For the first time, I experienced the difference between "the internet" and "the network."

Although external connectivity was unavailable, the lab itself continued operating. Active Directory, internal DNS, Docker services, BookStack, NexusAI, and communication between lab machines all remained available across the LAN.

The outage became less about losing internet access and more about discovering which parts of the system actually depended on it.

---

## AETHER Continued Operating

Even with the WAN connection unavailable, the local infrastructure remained functional.

Available services included:

* Active Directory
* Internal DNS
* Docker services
* BookStack documentation
* NexusAI
* Local virtual machines
* SSH and RDP between hosts
* Internal networking

This highlighted that AETHER has gradually evolved into an actual intranet rather than simply a collection of computers running services.

The internet had failed.

The lab had not.

---

## Discovering Another Dependency

The outage also exposed an important architectural dependency.

Hermes currently relies on cloud-hosted language models through OpenAI. Once internet access disappeared, Hermes effectively lost its reasoning capability.

This raised an important systems question:

**What should happen when cloud AI becomes unavailable?**

Rather than treating this as a failure, it became another dependency to map and eventually remove.

Current architecture:

Internet
→ Cloud LLM
→ Hermes

Desired architecture:

Internet available?

Yes
→ Cloud LLM

No
→ Local LLM

Both paths
→ Hermes

This would allow Hermes to gracefully degrade rather than becoming unavailable.

---

## First Local LLM Experiment

The outage provided an opportunity to begin experimenting with local language models.

Ollama had already been installed on Cass but had never been integrated with Hermes.

Several discoveries were made:

* Hermes Desktop can communicate with a local Ollama endpoint through a custom endpoint configuration.
* The older Llama 2 model successfully connected.
* Hermes recognized the local endpoint.
* The experiment exposed context window limitations and model quality issues rather than connectivity problems.

Although the resulting experience was not yet practical, the experiment successfully proved that local inference can be integrated into the existing workflow.

The limitation was not the architecture.

The limitation was the model.

---

## Future Local AI

Rather than continuing to tune Llama 2, the plan is to revisit local inference with newer open-weight models.

Possible future candidates include:

* Llama 3.x
* Qwen
* Mistral
* Gemma

The objective is not to replace cloud models entirely.

The objective is resilience.

If internet connectivity disappears, Hermes should continue functioning with reduced capability rather than failing completely.

---

## RustDesk Infrastructure

The outage also reinforced another long-term goal.

Today, RustDesk depends on public rendezvous and relay infrastructure.

Long-term, AETHER should provide its own RustDesk server so remote access remains under local control whenever practical.

Benefits include:

* Greater independence from public infrastructure
* More consistent remote management
* Better understanding of the remote desktop stack
* Another production service running inside AETHER

Like the local language model, this is less about avoiding cloud services and more about reducing unnecessary external dependencies.

---

## Systems Thinking

One realization from today's outage is that complexity naturally changes the kinds of questions being asked.

Early in the homelab, the primary question was:

"How do I make this work?"

As additional services were added, the question gradually shifted toward:

"What happens if this breaks?"

Eventually that became:

"What happens if something fundamental breaks?"

That shift feels significant.

It represents moving beyond simply deploying technology toward thinking about resilience, dependencies, failure modes, and recovery.

The questions arose naturally simply by living with the system.

---

## Current Status

Confirmed:

* Local infrastructure remains operational during WAN outage.
* AETHER functions as a true intranet.
* Hermes can communicate with a local Ollama endpoint.
* Local AI integration is technically viable.
* Current model quality is not yet sufficient for regular use.

Future work:

* Deploy a modern local language model once internet access returns.
* Continue evaluating Hermes local inference.
* Investigate hosting a private RustDesk server within AETHER.
* Continue identifying and documenting external dependencies.
* Expand disaster recovery documentation to include WAN outage scenarios.

---

## Lessons Learned

Today's outage reinforced that resilience is not achieved by avoiding failures.

It is achieved by understanding dependencies.

Every service depends on something else.

The more clearly those dependencies are understood, the easier it becomes to design graceful degradation instead of complete failure.

Ironically, losing internet access became one of the most valuable architectural exercises AETHER has experienced.

It revealed not what was broken, but what was already resilient.

---

## Closing Note

AETHER is gradually becoming more than a homelab.

It is becoming an environment capable of continuing useful work even when individual dependencies disappear.

Today's outage demonstrated that local infrastructure, documentation, authentication, messaging, and internal services already possess that resilience.

The next step is extending that philosophy to remote access and artificial intelligence so that the lab remains functional even when cloud services are unavailable.

Resilience is becoming one of AETHER's core design principles.
