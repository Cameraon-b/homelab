# Learned Monitoring Design Principles

Discovered an important observability concept:

## Avoid “Expected Red” Monitors

Initially considered monitoring services that would naturally go offline during dual-boot changes or normal behavior.

Realized this creates:

- alert fatigue
- noisy dashboards
- unreliable health indicators

Refined monitoring philosophy to:

> only monitor systems/services that are expected to remain consistently available

This resulted in a cleaner and more meaningful dashboard.

# DNS Troubleshooting & IPv6 Discovery

Initially encountered DNS failures when attempting to resolve:

- `kuma.aether.lab`

Discovered that client systems were still prioritizing:

- ISP-provided IPv6 DNS servers

instead of:

- AETHER internal DNS

This resulted in public DNS attempting to resolve internal AETHER records.

---

## Key Lesson Learned

Learned that:

- DNS server precedence matters
- IPv6 DNS can override intended IPv4 DNS behavior
- internal AD DNS environments require clients to use domain DNS servers

---

## Temporary Solution

On Reeba:

- manually configured IPv4 DNS to use `WIN-DC01`
- temporarily disabled IPv6 for testing and learning simplicity

Confirmed successful internal DNS resolution using:

```powershell
nslookup kuma.aether.lab
```

Successfully resolved:

```text
kuma.aether.lab → 192.168.1.137
```

---

# Successfully Routed Kuma Through Reverse Proxy

Configured Nginx Proxy Manager to forward:

```text
kuma.aether.lab
```

to:

```text
192.168.1.137:3001
```

Successfully accessed Kuma internally through browser using:

```text
http://kuma.aether.lab
```

instead of direct IP and port access.
