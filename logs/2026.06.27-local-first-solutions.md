## RustDesk Server on Nora

The first major success was deploying a self-hosted RustDesk server on Nora using Docker Compose.

RustDesk server components:

- `hbbs` — ID/rendezvous server
- `hbbr` — relay server

Nora was a natural host for this because it already runs core AETHER services and remains the primary always-on Linux server.

Both Cass and Reeba were configured to use Nora as the RustDesk ID and relay server.

Client settings used Nora’s LAN address:

```text
ID Server: 192.168.1.137
Relay Server: 192.168.1.137
API Server: blank
Key: Nora RustDesk public key
```

Both clients reported a successful connection.

This means Cass and Reeba can now register with AETHER’s own RustDesk server rather than relying on public RustDesk infrastructure.

A full WAN-disconnect test was intentionally postponed because the internet had only recently returned and I did not want to interrupt household internet access again.

Current status:

- RustDesk server is running on Nora.
- Cass and Reeba successfully connect to it.
- Full outage validation still needs to be performed.

The next goal was improving local AI beyond the initial Llama 2/Ollama experiment from Log #28.

A newer model was needed.

The limitation was the model.

## LM Studio Replaced Ollama for Experimentation

Ollama worked, but configuration with Hermes was awkward.

Hermes repeatedly complained about context length, and changing YAML/cache files did not reliably affect behavior.

LM Studio became the better tool for experimentation because it provides:

- A graphical model manager
- Local model loading controls
- Manual context and parameter settings
- An OpenAI-compatible local API server
- Easier model switching

LM Studio exposed a local endpoint:

```text
http://localhost:1234/v1
```

Hermes was then configured to use this endpoint as a custom API provider.

After discovering that context and model parameters must be set before starting the LM Studio server, Hermes successfully connected to a local Qwen model.

Working path:

```text
Hermes
  ↓
LM Studio local server
  ↓
Qwen
  ↓
Cass
```

This was the first practical local AI path for Hermes.

It is slower than ChatGPT, but it works.

That makes it useful as an outage fallback.

## Local Model Role Thinking

The session also clarified how different models may fit different AETHER roles.

Qwen feels better suited for Hermes because Hermes is more technical, direct, and engineering-oriented.

Gemma seemed like a good candidate for Mira because Mira’s role is documentation, explanation, and reflection.

This led to a useful shift in thinking:

The agent identity is not the model.

The agent identity is made from:

- Name
- Role
- Prompt
- Memory
- Responsibilities
- Tools
- Context
- Model choice

The model is only the reasoning engine underneath.

This means Hermes can move from ChatGPT to Qwen locally without losing the Hermes role, and Mira could potentially use Gemma without ceasing to be Mira.

## Hardware Limits

Local model testing also clarified the current hardware limits.

Cass appears best suited for models in the 7B to 14B range.

Important limits:

- 16 GB RAM is the primary constraint.
- The GTX 1650 SUPER has limited VRAM for modern local models.
- Larger context windows significantly increase memory use.
- 32B models may be technically possible in some cases, but are not likely to be practical on current hardware.

Current practical range:

```text
7B  — good daily local fallback
12B — possible, slower
14B — worth testing
32B — likely not worth it yet
```

Future local AI work may be better distributed across Cass and Reeba depending on Reeba’s specs.

## OpenClaw Integration Attempt

After Hermes was working with LM Studio, the next experiment was connecting OpenClaw to the same local model server.

This turned out to be much harder than Hermes.

A key discovery was that OpenClaw separates model configuration into two parts:

- Provider
- Model

The provider is the API endpoint.

The model is the named model available through that provider.

For LM Studio, the provider should point to:

```text
http://localhost:1234/v1
```

The model should reference whatever LM Studio is serving.

OpenClaw model references appear to combine provider key and model ID:

```text
provider-key/model-id
```

Example:

```text
lm-studio/google/gemma-4-e4b
```

This required the provider key and model reference to match exactly.

## OpenClaw Configuration Trouble

The OpenClaw configuration became unstable during local model testing.

Symptoms included:

- Deleted models still appearing in the chat model selector.
- Newly added Gemma model not appearing in the selector.
- Model settings saving in one place but not reflecting in chat.
- Plugin version drift warning for the `codex` plugin.
- Gateway restarts restoring unexpected state.
- Local model configuration overriding access to ChatGPT.

The `openclaw gateway status --deep` command revealed that OpenClaw was using:

```text
~\.openclaw\openclaw.json
```

It also showed plugin drift:

```text
codex: 2026.6.1 → expected 2026.6.10
```

The `codex` plugin was updated, which helped partially.

However, the model registry and chat selector still behaved inconsistently.

At one point, OpenClaw was configured to use Gemma as the primary model, but the provider/model reference did not resolve correctly.

This caused OpenClaw to fail with:

```text
Unknown model: lm-studio/google/gemma-4-e4b
```

OpenAI/ChatGPT access was then restored by changing the agent default back to:

```text
gpt-5.5
```

and changing the model configuration mode from:

```text
replace
```

to:

```text
merge
```

This allowed ChatGPT/OpenAI to function again while leaving room for local models later.

## Lessons Learned

The biggest lesson was that local AI is not one thing.

It is a stack.

```text
Model file
  ↓
Model runner
  ↓
API server
  ↓
Provider config
  ↓
Model registry
  ↓
Agent defaults
  ↓
Chat interface
```

Each layer can fail independently.

Hermes and LM Studio were relatively straightforward once the correct server parameters were loaded.

OpenClaw was more complex because its gateway, plugins, providers, models, agent defaults, and chat selector all appear to maintain related but separate pieces of state.

The second lesson was that working local AI should not replace a stable cloud workflow until the local path is proven.

ChatGPT/OpenAI remains the primary high-quality model path.

Local models are currently best treated as fallback, experimentation, and resilience infrastructure.

The third lesson was operational:

Always keep a backup of working configuration before editing an agent framework config file.
