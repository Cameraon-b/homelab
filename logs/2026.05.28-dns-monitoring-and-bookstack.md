---

### DNS Troubleshooting

Investigation revealed:

```text
Reeba
↓
WIN-DC01 DNS
↓
Successfully resolved aether.lab records
```

while:

```text
Nora
↓
Router DNS
↓
Failed to resolve aether.lab records
```

Testing confirmed:

```bash
nslookup wiki.aether.lab 192.168.1.15
```

successfully resolved records directly against the domain controller.

Further investigation using:

```bash
resolvectl status
resolvectl dns
resolvectl domain
nslookup
dig
```

identified that Nora was still utilizing DNS settings inherited from DHCP rather than relying solely on WIN-DC01.

---

### Netplan Cleanup

Discovered multiple active Netplan YAML files:

```text
00-installer-config.yaml
backup.yaml
```

The backup configuration was still being processed by Netplan.

Actions performed:

* Removed backup configuration from active Netplan processing
* Simplified bridge configuration
* Configured Nora to use WIN-DC01 as its primary DNS server
* Disabled DHCP-provided DNS settings
* Reduced DNS path complexity

Updated Netplan configuration:

```yaml
network:
  version: 2

  ethernets:
    enp2s0: {}

  bridges:
    br0:
      interfaces: [enp2s0]

      dhcp4: true
      dhcp6: false

      dhcp4-overrides:
        use-dns: false

      nameservers:
        addresses:
          - 192.168.1.15
```

Applied configuration:

```bash
sudo netplan generate
sudo netplan apply
```

## Lessons Learned

### DNS Is Infrastructure

Application monitoring failures may originate several layers below the application itself.

The observed failure path was:

```text
BookStack Monitor Failure
↓
DNS Resolution Failure
↓
Resolver Configuration
↓
DHCP DNS Inheritance
```

rather than any issue with BookStack itself.

---

### Internal DNS Requires Adoption

Creating DNS records alone is insufficient.

All infrastructure systems must be configured to use the authoritative DNS server.

Without proper DNS integration:

```text
Services Exist
DNS Records Exist
Applications Work

But Infrastructure Cannot Discover Them
```

---

### Monitoring DNS Names Is Superior

Monitoring:

```text
wiki.aether.lab
```

validates:

```text
DNS
↓
Reverse Proxy
↓
Application
```

whereas monitoring:

```text
192.168.1.137:6875
```

only validates:

```text
IP
↓
Port
```

Application-level monitoring provides significantly greater visibility into service health.
