# AETHER Log #31 — WAN Outage Validation and Protecting the Platform

**Date:** June 30, 2026

---

# Summary

Today's work focused on answering an important architectural question:

> **What happens if the internet disappears?**

Rather than continuing to add services, AETHER was intentionally placed into a simulated WAN outage by disconnecting the network connection to the modem while leaving the local network intact.

The goal was to validate whether AETHER had actually become a local-first environment, or whether many of its capabilities still depended on cloud services.

The results were encouraging.

Although cloud AI and public internet services became unavailable, the core infrastructure continued operating across the LAN.

BookStack, Homepage, Gitea, Docker services, Active Directory, internal DNS, and the self-hosted RustDesk server remained available. Local AI through LM Studio also continued functioning, providing a degraded but usable fallback for engineering assistance.

The exercise confirmed that AETHER is no longer simply a homelab hosted on a server—it is becoming an engineering environment capable of continuing operations during an external network failure.

---

# WAN Outage Simulation

The outage test was performed by disconnecting the WAN connection while leaving the router, switches, and local network powered.

This simulated a typical ISP outage without disrupting the internal LAN.

Communication with ChatGPT was maintained separately using an iPad connected through a cellular hotspot, providing an out-of-band management channel similar to emergency administrative access.

# Results

## Infrastructure

Confirmed operational during the outage:

- Homepage
- BookStack
- Gitea
- NexusAI
- Uptime Kuma
- Nginx Proxy Manager
- Active Directory
- Internal DNS
- Docker services

Internal service discovery through `home.aether.lab` continued functioning normally.

The Homepage proved especially valuable during the outage because it provided a single entry point to every internal service without relying on bookmarks or internet resources.

---

## RustDesk Validation

The self-hosted RustDesk server successfully demonstrated LAN-only operation.

An existing remote session remained connected briefly after the WAN connection was removed before disconnecting approximately one minute later.

After reopening RustDesk and confirming the configured ID/Relay server, reconnection succeeded over the local network.

Although the reconnect behavior warrants additional investigation, the overall objective was achieved:

**Remote administration remained possible without internet connectivity.**

---

## Local AI Validation

LM Studio continued functioning normally during the outage.

Hermes was able to communicate with locally hosted language models despite the loss of internet access.

Two local models were evaluated.

### Qwen 2.5 Coder

Provided quick and practical Docker troubleshooting guidance, including:

- Checking container status
- Viewing Docker logs
- Restarting services
- Verifying Docker operation

### Gemma

Produced a significantly slower response but generated a more detailed troubleshooting workflow focused on diagnosis before recovery.

The practical takeaway was that local AI does not replace cloud models but provides a useful operational fallback.

Engineering assistance remained available even when OpenAI services were unreachable.

# Architectural Observation

The outage highlighted an important distinction.

Before this project, losing internet connectivity largely meant losing the ability to work.

After today's validation, losing internet primarily meant losing external resources.

The engineering environment itself remained operational.

This represents an important architectural shift.

AETHER now provides continuity of operations rather than depending entirely on cloud infrastructure.

---

# Documentation and Publishing Workflow

Another significant design discussion focused on documentation.

BookStack continues to serve as AETHER's operational knowledge base.

Rather than maintaining separate documentation systems, the intended workflow is now:

```text
BookStack
        ↓
Mira
(sanitize and convert)
        ↓
Markdown
        ↓
Local Git Repository
        ↓
Gitea
        ↓
GitHub
```

This establishes BookStack as the authoritative source of operational knowledge while allowing a curated, public-safe version of the documentation to be published as part of the homelab repository.

This approach avoids maintaining two independent sets of documentation and keeps public documentation closely aligned with the operational knowledge used to manage the lab.

# Backup Planning

Today's outage test also changed the perceived value of the platform.

Earlier backups primarily protected virtual machines.

Now the rapidly changing services themselves represent the most valuable part of AETHER.

Current priorities include backing up:

- BookStack
- Gitea
- Homepage
- NexusAI
- Uptime Kuma
- Memos
- Nginx Proxy Manager
- Docker Compose projects
- Configuration files
- `/etc`

Virtual machine images will continue to be backed up less frequently since their configuration changes relatively little compared to the Docker services.

An automated maintenance script is planned to perform nightly backups, verify the backup drive is mounted, rotate older backups, and generate backup manifests.

---

# Lessons Learned

The most important lesson was that resilience is about graceful degradation rather than perfection.

Cloud AI disappeared during the outage.

Local AI remained available.

GitHub disappeared.

Gitea remained available.

The internet disappeared.

The engineering environment continued functioning.

The second lesson was that infrastructure eventually becomes valuable enough to justify operational processes.

As AETHER has grown, backups, documentation, maintenance, and validation have become just as important as deploying new services.

The project is gradually shifting from building infrastructure to operating it

# Closing Note

Today's work demonstrated that AETHER can continue supporting meaningful engineering work even when disconnected from the internet.

The outage did not stop development.

It merely changed which tools were available.

That distinction feels significant.

For the first time, AETHER behaved less like a collection of self-hosted services and more like an environment designed to keep working when external dependencies fail.

The test also revealed something equally important: AETHER has reached the point where its knowledge and workflows are becoming more valuable than the hardware itself. Protecting BookStack, Gitea, NexusAI, and the growing collection of self-hosted services is now a higher priority than simply adding new infrastructure.

The project is entering a new phase.

Less time will be spent asking, *"What should I build next?"*

More time will be spent asking, *"How do I operate, protect, document, and improve what I've already built?"*

That feels like an important milestone.
