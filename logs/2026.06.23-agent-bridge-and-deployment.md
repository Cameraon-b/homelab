# AETHER Log #27 — NexusAI Agent Bridge and Verified Nora Deployment

**Date:** June 23, 2026

## Summary

Today marked a major stabilization milestone for NexusAI and the broader AETHER Lab. The main accomplishment was proving that the NexusAI agent bridge now works end-to-end for both Hermes and Mira, while also fixing and verifying the Nora deployment workflow so the running service can be tied back to an actual GitHub commit.

What started as another round of confusing bridge testing turned into a much clearer architecture:

* NexusAI is the coordination room.
* Workers are the couriers.
* Bridge responders are the agent voice.
* Cameron remains the approval gate.

The system is still intentionally non-executing. Agents can draft and route replies through the bridge, but they cannot run commands, restart services, deploy code from the UI, or bypass approval.

## Agent Bridge v1 Confirmed Working

The bridge-file architecture was manually tested and confirmed for both agents.

Confirmed flow:

1. A message is sent through NexusAI.
2. Cameron approves AI-to-AI delivery.
3. The receiving agent worker checks the inbox.
4. The worker creates a bridge request JSON file.
5. The bridge responder reads the request file.
6. The bridge responder generates a response JSON file.
7. The worker consumes the response file.
8. The worker posts the reply back to NexusAI.
9. The reply waits for Cameron approval before delivery.
10. The recipient receives the approved reply.

Both directions were verified:

* Mira → Hermes → Mira
* Hermes → Mira → Hermes

Confirmed working:

* Request file creation
* Response file generation
* Response posting
* Correct routing
* Duplicate prevention
* Approval-gated delivery
* No command execution

This establishes Agent Bridge v1 as working in template responder mode.

## Bridge Architecture Clarified

The project now has a clearer separation of responsibilities.

### NexusAI

NexusAI stores messages, conversations, approvals, tasks, notices, review requests, status records, and logs.

NexusAI does not execute commands.

### Agent Worker

The worker acts as a courier. It checks NexusAI for delivered messages, creates bridge request files, consumes bridge response files, posts replies back to NexusAI, and exits.

The worker is not the agent brain.

### Bridge Responder

The responder reads request JSON files and writes response JSON files.

This allows the reply generation layer to evolve independently. Today it works in template mode, but the same boundary could later support OpenClaw, a local model, a hosted model, or a manual response workflow.

### Cameron Approval Gate

AI-to-AI communication still requires Cameron approval before delivery. The bridge does not bypass approval.

## Deployment Issue Found and Fixed

A major deployment confusion was identified. GitHub contained newer NexusAI commits, but Nora was still running an older commit.

Nora was on:

```text
83089b9
```

GitHub origin/main was on:

```text
60e881c
```

This explained why the version tracker was not visible in the running NexusAI UI. The feature existed in GitHub, but Nora had not actually pulled and deployed the relevant commits.

After manually pulling the latest code and running the deploy process on Nora, the version information appeared correctly.

## GitHub Actions Deployment Improved

The GitHub Actions deploy workflow was updated so the Nora runner explicitly syncs the local repository to `origin/main` before deploying.

The workflow now performs a stronger deployment sequence:

1. Show runner identity.
2. Change into `/home/noratheredeemer/nexusai`.
3. Fetch `origin/main`.
4. Reset Nora’s working tree to match GitHub.
5. Run `deploy.sh`.
6. Verify Docker Compose status.
7. Query the version endpoint.

This fixes the previous ambiguity where GitHub Actions could appear to run but Nora might still be behind.

The version display now provides a direct way to verify what code is actually running.

## Bridge-Cycle Scripts Added

Hermes added bridge-cycle wrapper scripts for both agents.

Planned scripts:

```text
scripts/run-hermes-bridge-cycle.ps1
scripts/run-mira-bridge-cycle.ps1
```

These wrappers run the full bridge cycle:

```text
worker creates request
→ responder generates response
→ worker posts response
```

Mira’s bridge-cycle script was tested successfully on Reeba.

Hermes’ script on Cass initially failed because Windows PowerShell script execution was disabled on that machine. This was not a NexusAI failure. It was a local Windows execution policy issue.

The workaround is to run the script with:

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\scripts\run-hermes-bridge-cycle.ps1
```

For Task Scheduler, the same execution policy bypass should be used in the task arguments.

## Important Safety Boundary

NexusAI remains a coordination, messaging, bridge, approval, and audit system.

The following remain intentionally outside the boundary:

* Shell execution
* PowerShell execution
* SSH execution
* Docker control from the UI
* Service restarts from the UI
* Automatic remediation
* Automatic approvals
* Secret handling
* Autonomous infrastructure changes

AETHER rule remains:

```text
No agent shall chown the kingdom.
```

## Lessons Learned

The major lesson from this phase was that the system needed a clearer mental model.

The project felt stuck because several layers were being tested repeatedly without clearly naming which layer was under test:

* NexusAI message storage
* AI-to-AI approval
* Worker polling
* Bridge request creation
* Response generation
* Response posting
* Routing
* Deployment
* Version verification

Once these layers were separated, the state of the system became much clearer.

The second lesson was that a deployment pipeline needs proof. A successful workflow run is not enough. The running application needs to expose what commit it is actually running.

The version endpoint and UI version display now provide that proof.
