# Internal DNS Troubleshooting

Discovered that internal DNS resolution only worked on systems configured to use WIN-DC01 as their DNS server.

Observed behavior:

* Reeba successfully resolved:

  * `kuma.aether.lab`
  * `wiki.aether.lab`
* Cass could not resolve internal domains because it was still using the router/Spectrum DNS configuration.

This reinforced understanding of:

* internal DNS architecture
* client DNS configuration
* domain-specific resolution
* separation between public DNS and internal infrastructure DNS

IPv6 DNS behavior also interfered with initial testing. Reeba required temporary IPv6 disablement to force usage of the Windows Server DNS infrastructure.

# VM Instability Investigation

Observed repeated outages in Uptime Kuma where:

```text
WIN-DC01
WIN-CLIENT01
```

would unexpectedly go offline.

Using:

```bash
virsh list --all
```

revealed both VMs had entered a paused state.

Initial suspicion included:

* memory exhaustion
* virtualization instability
* hypervisor resource contention

However memory investigation showed:

```bash
free -h
```

still reported approximately 4GB available.

Further investigation using:

```bash
journalctl -xe
```

revealed repeated:

```text
No space left on device
```

errors from rsyslog.

---

# Filesystem Exhaustion Diagnosis

Using:

```bash
df -h
```

revealed:

```text
/dev/mapper/ubuntu--vg-ubuntu--lv
100% full
```

Further storage investigation showed:

```bash
lsblk
```

revealed a 1TB physical drive while the active logical volume was only approximately 100GB.

Using:

```bash
vgs
```

revealed over 800GB of unused free space inside the LVM volume group.

This became a major learning moment involving:

* LVM architecture
* logical volumes
* volume groups
* filesystem expansion
* infrastructure capacity planning

---

# Live LVM Expansion

Expanded the live filesystem without rebooting the server.

Commands used:

```bash
sudo lvextend -l +100%FREE /dev/ubuntu-vg/ubuntu-lv
sudo resize2fs /dev/ubuntu-vg/ubuntu-lv
```

Result:

```text
Filesystem expanded from ~100GB to ~915GB usable
```

Final validation:

```bash
df -h
```

showed:

```text
915G total
784G available
11% used
```

After expansion:

* VM stability returned
* Kuma services remained online
* logging resumed normally
* no additional pause events observed

This was one of the most significant infrastructure troubleshooting and recovery exercises completed so far in the homelab.

# OpenClaw / Mira Gateway Incident

Experienced the first significant OpenClaw gateway incident on Reeba while modifying appearance settings inside the dashboard.

Symptoms included:

* dashboard freezing
* disappearing terminal windows
* websocket disconnects
* inability to reconnect to the dashboard
* gateway probe timeouts

Initial diagnostics revealed:

* port 18789 already bound
* Node process still partially active
* websocket probe failures
* stale or hung gateway state

Troubleshooting steps included:

```powershell
openclaw gateway status --deep
openclaw gateway stop
taskkill /PID /F
openclaw gateway start
openclaw gateway run
```

An attempted PID termination revealed the process had already exited, indicating stale service state rather than an actively hung process.

Running the gateway in foreground/debug mode exposed additional service behavior and confirmed:

* loopback binding functionality
* scheduled task integrity
* dashboard websocket dependency
* successful gateway recovery

Ultimately the gateway restarted successfully and the dashboard became operational again.

This became the first real “production incident” involving the OpenClaw environment and provided practical troubleshooting experience involving:

* ports
* processes
* websocket services
* background task management
* service recovery workflows
