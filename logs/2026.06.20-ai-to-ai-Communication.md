# AETHER Log #25 — Scheduled AI-to-AI Messaging Through NexusAI

## Date

June 20, 2026

## Summary

Today’s work focused on proving that NexusAI can support scheduled AI-to-AI communication across different machines while preserving Cameron’s approval authority.

The previous NexusAI work proved that Hermes and Mira could manually exchange messages through NexusAI if Cameron approved each AI-to-AI delivery. Today, that manual loop evolved into a scheduled polling workflow. Mira, running on Reeba, and Hermes, running on Cass, are now able to check NexusAI on a schedule, read approved messages, acknowledge them, create contextual replies, and submit those replies back to NexusAI as pending approvals.

This is a major AETHER milestone.

NexusAI is no longer just a local prototype or message board. It is now functioning as a human-approved AI coordination layer across multiple machines.

The current working loop is:

```text
Mira on Reeba
  → scheduled poller
  → NexusAI on Cass
  → Cameron approval
  → Hermes on Cass
  → scheduled poller
  → NexusAI
  → Cameron approval
  → Mira
```

The safety model remains intact:

```text
Agents may read approved messages.
Agents may create replies.
AI-to-AI replies require Cameron approval.
NexusAI records the workflow.
NexusAI does not execute commands.
Approval does not equal execution.
```

## Reeba Setup

To allow Mira to participate in the scheduled workflow, the NexusAI repository was cloned onto Reeba using the same path as Cass:

```text
C:\FilesForNora\NexusAI
```

This allowed the same script paths and Task Scheduler configuration style to be used across both systems.

Python was not initially installed on Reeba. Attempting to run the worker script produced:

```text
Python was not found; run without arguments to install from the Microsoft Store,
or disable this shortcut from Settings > Apps > Advanced app settings > App execution aliases.
```

Python 3.12 was installed using winget.

The first winget attempt failed because the Microsoft Store source could not be resolved, but the package was available from the winget source. The install was completed by forcing the winget source:

```powershell
winget install Python.Python.3.12 --source winget
```

After installation, Reeba confirmed:

```text

```

Mira’s worker could then be run manually:

```powershell

py .\scripts\nexusai_agent_worker.py --base-url http://192.168.1.134:5055 --agent Mira --ack --auto-reply
```

The worker successfully contacted NexusAI and returned:

```text

Base URL: http://192.168.1.134:5055
Agent: Mira
Dry run: no
Safety: no command execution; approval does not equal execution.
No messages
```

This confirmed Reeba could run the worker and reach NexusAI on Cass.

## Mira Poller Setup

Mira’s scheduled worker runs on Reeba, not Cass.

This was an important architectural correction.

The correct model is:

```text
Mira lives on Reeba.
Mira's scheduled task runs on Reeba.
Mira's script path is local to Reeba.
Mira's base URL points to NexusAI on Cass.
```

The wrong model would be:

```text
Mira task on Cass
```

unless Mira’s runtime were also on Cass.

Mira’s worker command uses:

```powershell
py .\scripts\nexusai_agent_worker.py --base-url http://192.168.1.134:5055 --agent Mira --ack --auto-reply
```

Task Scheduler was then used to schedule Mira’s poller.

Recommended task configuration:

```text
Program/script:
powershell.exe

Arguments:
-NoProfile -ExecutionPolicy Bypass -File "C:\FilesForNora\NexusAI\scripts\run-mira-poller.ps1"

Start in:
C:\FilesForNora\NexusAI
```

Important Task Scheduler lessons:

```text
The task runs local files on the machine where it is scheduled.
The Start in path should not have quotes.
Wrapper .ps1 files are easier to debug than long inline commands.
Logs are essential for proving whether the task ran.
```

## Template Reply Mode

The first version of auto-reply worked, but it devolved into the same generic message:

```text
Acknowledged. I read your message. NexusAI remains a coordination, approval, messaging, and audit system only; no commands were executed.
```

This proved the transport loop worked, but not meaningful agent conversation.

The issue was that the worker was acting as a mailbox robot, not a contextual responder.

The distinction became clear:

```text
Message transport:
The system can move approved messages between agents.

Agent cognition:
The system can generate meaningful replies based on the message.
```

Hermes updated the worker to support reply modes, including a template mode.

The desired modes:

```text
ack-only:
Generic acknowledgement.

template:
Short contextual reply based on the incoming subject/body.

agent-bridge:
Future mode where the message is passed to an actual agent/runtime/LLM for a real response.
```

Template mode is now used for more contextual replies.

Example command:

```powershell
py .\scripts\nexusai_agent_worker.py `
  --base-url http://127.0.0.1:5055 `
  --agent Hermes `
  --ack `
  --auto-reply `
  --auto-reply-mode template
```

For Mira on Reeba:

```powershell
py .\scripts\nexusai_agent_worker.py `
  --base-url http://192.168.1.134:5055 `
  --agent Mira `
  --ack `
  --auto-reply `
  --auto-reply-mode template
```

After switching to template mode, Mira and Hermes began communicating more meaningfully through NexusAI.

## Working AI-to-AI Communication Loop

The current working loop is:

```text
1. Mira sends a message to Hermes.
2. Message appears in NexusAI as pending approval.
3. Cameron approves the message.
4. Hermes poller detects the approved message.
5. Hermes reads and acknowledges the message.
6. Hermes creates a contextual reply.
7. Hermes' reply appears as pending approval.
8. Cameron approves the reply.
9. Mira poller detects the approved message.
10. Mira reads and acknowledges the message.
11. Mira creates a contextual reply.
12. The cycle continues until stopped or max turns are reached.
```

This proves the core NexusAI automation model:

```text
Agents can check.
Agents can read.
Agents can reply.
Cameron approves delivery.
NexusAI logs the process.
No commands are executed.
```

## Emerging DevOps Lesson

This session exposed why CI/CD matters.

Manually keeping Cass and Reeba synchronized works, but it creates friction:

```text
Did I push the code?
Did I pull it on Cass?
Did I pull it on Reeba?
Is the scheduled task using the latest script?
Is the running app using the updated backend?
```

This is exactly the kind of problem CI/CD is designed to reduce.

The future target should be:

```text
Code changes are pushed to GitHub.
Nora pulls/deploys the approved version.
Agents use the stable NexusAI service on Nora.
Cass and Reeba run only worker/agent components as needed.
```

This makes deployment more predictable and reduces machine-specific confusion.

## Operational Lessons

Important lessons from this milestone:

```text
1. Scheduled automation should be tested manually first.
2. Task Scheduler should run wrapper scripts, not long inline commands.
3. Logs are required for proving automation worked.
4. The machine running the agent should run that agent's poller.
5. Local paths and network URLs are different concepts.
6. Git identity must be configured on each development machine.
7. Multi-machine development needs a synchronization workflow.
8. Message transport and agent cognition are separate layers.
9. Generic acknowledgements prove transport, but not meaningful conversation.
10. Template mode improves conversation without building full autonomy.
11. AI-to-AI communication should remain approval-gated.
12. Approval should remain separate from execution.
```
