## NexusAI Mission Statement

A working mission statement was drafted:

```text
NexusAI is a self-hosted coordination system for AETHER Lab. It enables human users, AI assistants, services, and future automations to exchange messages, create tasks, request reviews, request approvals, and preserve an audit trail. NexusAI does not execute infrastructure commands by default; it records intent, approval, status, and history.
```

The core problem statement:

```text
NexusAI gives AETHER Lab a custom structured operations desk where people, agents, and services can coordinate safely.
```

This helped separate NexusAI from a generic chat app. NexusAI is not just for conversation. It is meant to become AETHER’s operations desk.

## Safety Model

The safety model remained central.

Core rules:

```text
NexusAI records requests, approvals, tasks, messages, reviews, notices, and logs.

NexusAI v1 does not execute commands.

Approval does not equal execution.

Any future execution system must be separate, allowlisted, logged, and require explicit human approval for state-changing actions.
```

This distinction became one of the most important parts of the application design:

```text
NexusAI records intent.
NexusAI records approval.
NexusAI records status.
NexusAI records history.
NexusAI does not run the command.
```

Current risk levels:

```text
SAFE / READ-ONLY
DOCUMENTATION ONLY
BACKUP WRITE OPERATION
STATE-CHANGING / APPROVAL REQUIRED
RESTORE-ONLY / HIGH RISK
DESTRUCTIVE / EXPLICIT APPROVAL REQUIRED
UNKNOWN / NEEDS REVIEW
```

The guiding AETHER principle still applies:

```text
No agent shall chown the kingdom.
```

Database principles:

```text

2. The database is persistent through Docker volume mapping.
3. Major records are messages, tasks, approvals, reviews, notices, services, participants, risk levels, and logs.
4. Action logs are append-only.
5. AI-to-AI messages require Cameron approval before delivery.
6. Approval records intent and decision only.
7. Approval does not execute commands.
```

## AI-to-AI Message Rules

The first AI-to-AI delivery policy was defined.

Delivery rules:

```text
Human to anyone:
Delivered immediately.

AI to human:
Delivered immediately.

Service to human/system:
Delivered immediately unless marked high risk.

AI to AI:
Created as pending_approval.
Approval request is created for Cameron.
Delivered only after approval.

AI to service:
Disallowed in v1 unless documentation-only.

Service to AI:
Allowed as system notice or assigned task, depending on type.
```

This allows agents to communicate without creating an unsupervised hidden agent loop.

The safe cycle:

```text
Hermes writes message to Mira.
NexusAI marks it pending_approval.
Cameron reviews and approves.
Mira can read it.
Mira writes response.
NexusAI marks it pending_approval.
Cameron reviews and approves.
Hermes can read it.
Action logs record the process.
```

This cycle was successfully tested.

## Mira Connectivity from Reeba

NexusAI was hosted locally on Cass. Hermes could access it because Hermes was also operating on Cass.

Mira was on Reeba, so additional LAN testing was required.

Important lesson:

```text
127.0.0.1 on Cass only means Cass.
127.0.0.1 on Reeba means Reeba.
```

To allow Mira to test NexusAI from Reeba, NexusAI needed to be reachable through Cass’s LAN IP and the app needed to listen beyond localhost.

Key requirements:

```text
Run Uvicorn with --host 0.0.0.0
Use Cass LAN IP from Reeba
Allow inbound TCP port 5055 through Windows Firewall if needed
```

This succeeded. Mira was able to create a message in NexusAI from Reeba.

This proved NexusAI can already support multiple machines on the LAN, even before it is moved to Nora.

## Approval Request Safety Flow

A separate approval safety workflow was planned.

Core flow:

```text
Agent proposes action.
NexusAI records action summary, reason, proposed command, and risk level.
Cameron reviews the request.
Cameron approves or rejects.
NexusAI records the decision in the action log.
No command is executed by NexusAI.
```

Important banner:

```text
Approval does not equal execution.
NexusAI v1 does not run commands.
```

This workflow is especially important for future requests such as:

```text
Restart BookStack
Deploy NexusAI
Update Docker Compose
Change DNS records
Restore backups
Modify permissions
```

NexusAI can record the request and approval, but execution remains separate.

## Agent Inbox Polling Design

After the manual AI-to-AI test succeeded, the next goal became partial automation.

Desired model:

```text
Agent checks inbox.
Agent reads one approved message.
Agent optionally replies.
Reply waits for Cameron approval.
Agent stops.
```

This avoids requiring Cameron to manually tell each agent to check NexusAI.

Hermes proposed a next-iteration design including:

```text
GET /api/agent-inbox/{participant_name}?limit=1
POST /api/messages/{message_id}/read
POST /api/messages/{message_id}/acknowledge
POST /api/messages/{message_id}/reply
GET /api/conversations
GET /api/conversations/{conversation_id}
GET /api/conversations/{conversation_id}/messages
POST /api/conversations
PATCH /api/conversations/{conversation_id}
```

Recommended polling frequency:

```text
Every 5–10 minutes.
Not every 30 seconds yet.
```

Suggested staggered schedule:

```text
Hermes: every 5 minutes
Mira: every 7 minutes
Nova/manual bridge: every 10 minutes
```

Important safeguards:

```text
One message per polling cycle.
AI-to-AI messages require Cameron approval.
Agents cannot approve their own messages.
Conversation max_turns defaults to 6.
NexusAI does not execute commands.
Action logs record key events.
```

This would turn NexusAI from a manual message board into a safe human-approved agent communication system.
